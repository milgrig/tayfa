# Список сотрудников

## Единый реестр

Источник правды — **`.tayfa/common/employees.json`**. Оркестратор показывает **только** зарегистрированных там.

- Новые сотрудники — **только** через `python .tayfa/hr/create_employee.py <имя>`.
- Просмотр: `python .tayfa/common/employee_manager.py list`
- Ручная регистрация: `python .tayfa/common/employee_manager.py register <имя> "<роль>"`
- Удаление: `python .tayfa/common/employee_manager.py remove <имя>` (boss и hr удалить нельзя).

---

## boss

**Роль**: Руководитель, координатор всех задач.

**Когда обращаться к boss**:
- Задача выполнена — нужна следующая задача или подтверждение.
- Серьёзные вопросы по задаче.
- Нужно создать нового сотрудника.

**Кто может обращаться**: любой сотрудник.

---

## hr

**Роль**: HR-менеджер, управление сотрудниками и агентами.

**Что умеет hr**:
- Создавать новых сотрудников (онбординг) через `.tayfa/hr/create_employee.py`.
- Менять системные промты сотрудников.

**Кто может обращаться**: только `boss`.

---

## process_architect

**Роль**: Специалист по процессам и AI-агентам.

**Что умеет process_architect**:
- Анализировать и проектировать процессы разработки
- Оптимизировать работу команды AI-агентов
- Исследовать best practices в multi-agent системах
- Консультировать по архитектуре агентов и промтам
- Улучшать систему задач и оркестратор

**Когда обращаться к process_architect**:
- Нужно оптимизировать процессы команды
- Требуется экспертиза по AI-агентам и LLM
- Проблемы с эффективностью workflow
- Нужны рекомендации по структуре промтов или ролей
- Требуется исследование best practices

**Кто может обращаться**: boss, а также другие сотрудники через систему задач (customer/developer/tester).

---

## analyst

**Роль**: Системный аналитик, детализация требований и постановка задач.

**Что умеет analyst**:
- Анализировать требования от заказчика (boss)
- Детализировать ТЗ для разработчиков
- Писать user stories и acceptance criteria
- Декомпозировать задачи на подзадачи
- Принимать результаты (роль customer в задачах)
- Документировать функциональные требования

**Когда обращаться к analyst**:
- Нужно детализировать требования для фичи
- Требуется разбить крупную задачу на подзадачи
- Нужны user stories для нового функционала
- Требуется написать ТЗ для разработчика
- Готов результат для приёмки customer'ом

**Кто может обращаться**: boss (постановка задач), developer (уточнения по ТЗ), через систему задач.

---

## developer

**Роль**: Разработчик приложения, реализует функционал по ТЗ.

**Что умеет developer**:
- Писать код по техническим заданиям (Python, JavaScript/TypeScript)
- Реализовывать новые фичи и API endpoints
- Исправлять баги
- Рефакторить и оптимизировать код
- Писать тесты (unit, integration)
- Проводить code review

**Когда обращаться к developer**:
- Нужна реализация фичи по готовому ТЗ
- Нужно исправить баг
- Требуется рефакторинг кода
- Нужны тесты для функционала

**Роль в системе задач**: developer (разработчик)
- Получает задачи от analyst (customer) с детальным ТЗ
- Реализует функционал
- Отправляет результат на проверку tester

**Кто может обращаться**: analyst (через задачи), boss (напрямую), tester (уточнения).

---

## tester

**Роль**: Tester (QA Engineer) — тестировщик, проверка качества.

**Что умеет tester**:
- Тестировать реализованный функционал
- Проверять соответствие ТЗ и acceptance criteria
- Писать тест-кейсы и чек-листы
- Проводить регрессионное тестирование
- Документировать баги с шагами воспроизведения

**Когда обращаться к tester**:
- Задача разработана и готова к проверке (статус `на_проверке`)
- Нужна валидация багфикса
- Требуется регрессионное тестирование после изменений
- Нужно создать тест-кейсы для функционала

**Кто может обращаться**: analyst (назначает tester в задачах), developer (передаёт на проверку), boss (срочные проверки).

---

## python_dev

**Роль**: Senior Python Developer (Opus) — эксперт Python-разработки для сложных архитектурных задач.

**Что умеет python_dev**:
- Архитектурное проектирование и разработка core-модулей оркестратора Tayfa
- Глубокий рефакторинг больших компонентов системы (app.py, task_manager.py, и т.д.)
- Сложные интеграции (Claude API, Git automation, subprocess)
- Оптимизация производительности критических участков кода
- Применение Design Patterns, SOLID, Clean Architecture
- Полная типизация (type hints), docstrings, pytest-тесты

**Когда обращаться к python_dev**:
- Архитектурные изменения в core-модулях Tayfa
- Сложный рефакторинг больших файлов (>300 строк)
- Критические баги, требующие глубокого анализа
- Разработка новых подсистем (плагины, кэширование, streaming)
- Оптимизация производительности
- Задачи, требующие экспертизы Python (asyncio, metaclasses, protocols)

**Роль в системе задач**: developer (разработчик)
- Получает задачи от analyst (customer) с детальным ТЗ
- Реализует архитектурные решения высокого качества
- Отправляет результат на проверку tester

**Модель**: Claude Opus (claude-opus-4-5-20251101) — для сложных задач, где важно качество архитектуры.

**Кто может обращаться**: boss (постановка задач), analyst (через задачи), другие разработчики (консультации по архитектуре).

**Когда НЕ использовать python_dev** (используй developer на sonnet):
- Простые CRUD endpoints
- Добавление полей в JSON/модели
- Исправление опечаток
- Простой рефакторинг (переименование)
- Написание unit-тестов для готового кода

## Система задач

Каждая задача имеет **3 исполнителей**: заказчик, разработчик, тестировщик. Задачи создаёт boss через `.tayfa/common/task_manager.py`. Подробнее — `.tayfa/common/Rules/teamwork.md`.

```bash
python .tayfa/common/task_manager.py list              # все задачи
python .tayfa/common/task_manager.py get T001          # одна задача
python .tayfa/common/task_manager.py result T001 "..." # записать результат
python .tayfa/common/task_manager.py status T001 ...   # изменить статус
```

## Правила обращения

1. Не знаешь к кому — обращайся к `boss`.
2. К `hr` напрямую не обращайся (если ты не `boss`).
3. Для обращения — JSON в `request/`:

```json
{
  "name": "boss",
  "prompt": "Описание вопроса или результата"
}
```
